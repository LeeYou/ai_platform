## 1. 文档目的

本文档面向第三方调用者，说明当前 `ai_platform` 生产环境已提供的对外 REST API，用于：

- 查询服务健康状态
- 查询当前已加载能力
- 查询许可证状态
- 调用指定 AI capability 做推理
- 调用 agface 人脸比对组合接口

本文档以当前 `prod/web_service/main.py` 中已实现的真实接口为准。

## 3. 通用返回结构

## 3.1 通用推理成功返回

`POST /api/v1/infer/{capability}` 成功时统一返回：

```json
{
  "code": 0,
  "message": "success",
  "capability": "agface_face_detect",
  "model_version": "1.0.0",
  "inference_time_ms": 12.34,
  "result": {},
  "timestamp": "2026-04-23T16:00:00+08:00"
}
```

说明：

- `code = 0` 表示成功
- `result` 内部结构由具体 capability 决定
- `model_version` 来自 runtime 当前已加载的模型版本

## 3.2 通用失败返回

常见失败响应：

```json
{
  "code": 4004,
  "message": "Capability version not licensed",
  "capability": "agface_fake_photo"
}
```

或：

```json
{
  "code": 3001,
  "message": "Instance pool timeout or capability not available",
  "capability": "face_detect"
}
```

说明：

- HTTP 状态码与 `code` 一起判断
- `code` 为平台业务错误码
- `message` 为可读错误说明

## 5.2 应用层翻拍照检测

### 接口

```http
POST /api/v1/infer/detect_fake_photo
```

### 接口说明

该接口是面向第三方客户提供的**翻拍照综合检测接口**。

它不是单一模型接口，而是服务端封装好的应用层编排接口。当前代码中的执行顺序如下：

1. 先调用 `agface_fake_photo`
2. 若 `agface_fake_photo` 判定结果为“真人图像”，则继续调用 `desktop_recapture_detect`
3. 服务端综合上述结果，返回：
   - 最终判定结果
   - 判定来源
   - 最大人脸占图比例
   - 两个子能力的完整结果

因此，第三方客户无需分别调用多个底层 capability，只需调用本接口即可获得完整检测结果。

### Form 参数

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `image` | file | 是 | 待检测图片，建议使用清晰的 JPG/PNG 正脸或近似正脸图像 |

### 处理逻辑说明

#### 第一阶段：`agface_fake_photo`

该阶段优先进行 agface 体系内的翻拍/假体判断。

在当前实现中：

- `agface_fake_photo.result.result == 1`
  - 表示该子能力已判定为**翻拍/假体**
  - 此时接口直接以该子能力作为最终结论
  - 不再调用 `desktop_recapture_detect`

- `agface_fake_photo.result.result == 0`
  - 表示该子能力当前判定为**真人图像**
  - 此时服务端继续调用 `desktop_recapture_detect` 做第二阶段判断

- 如果 `agface_fake_photo` 未检测到人脸
  - 当前代码会将最终标签标记为 `no_face`
  - 第二阶段不会执行

#### 第二阶段：`desktop_recapture_detect`

只有在第一阶段判定为真人时，才会执行该阶段。

该子能力主要输出：

- `is_fake`
  - 是否判定为翻拍图
- `label`
  - 文本标签，通常为 `real` 或 `fake`
- `score_real`
  - 真人分数
- `score_fake`
  - 翻拍分数

### 返回结果总体说明

返回体外层仍遵循平台统一格式：

- `code = 0` 表示接口执行成功
- `capability = "detect_fake_photo"`
- `model_version = "composite"`
  - 表示该接口是应用层组合接口，不对应单一模型版本

### 请求示例

```bash
curl -X POST "http://localhost:8080/api/v1/infer/detect_fake_photo" \
  -F "image=@test.jpg"
```

### 成功返回示例

```json
{
  "code": 0,
  "message": "success",
  "capability": "detect_fake_photo",
  "model_version": "composite",
  "inference_time_ms": 26.41,
  "result": {
    "final_is_fake": false,
    "final_label": "real",
    "decision_source": "desktop_recapture_detect",
    "max_face_area_ratio": 0.218437,
    "agface_fake_photo": {
      "capability": "agface_fake_photo",
      "model_version": "1.0.0",
      "error_code": 0,
      "error_msg": "",
      "result": {
        "status": 1,
        "result": 0,
        "confidence": 0.2143,
        "face_bbox": [120, 80, 220, 220]
      }
    },
    "desktop_recapture_detect": {
      "capability": "desktop_recapture_detect",
      "model_version": "1.0.0",
      "error_code": 0,
      "error_msg": "",
      "result": {
        "is_fake": false,
        "label": "real",
        "score_real": 0.9211,
        "score_fake": 0.0789,
        "execution_provider": "cpu"
      }
    },
    "desktop_recapture_executed": true,
    "desktop_recapture_skip_reason": null
  },
  "timestamp": "2026-04-23T18:40:00+08:00"
}
```

### 返回字段说明

#### 顶层 `result` 字段

| 字段 | 类型 | 含义 |
|---|---|---|
| `final_is_fake` | `boolean \| null` | 最终是否判定为翻拍/假体。`true` 表示最终判定为翻拍，`false` 表示最终判定为真人，`null` 表示当前请求没有形成明确真假结论，例如未检测到人脸。 |
| `final_label` | `string` | 最终标签。当前实现下常见值为：`fake`、`real`、`no_face`、`unknown`。第三方客户应以该字段作为首选可读结论。 |
| `decision_source` | `string \| null` | 最终判定来源。若值为 `agface_fake_photo`，表示第一阶段已直接给出最终结论；若值为 `desktop_recapture_detect`，表示最终结论来自第二阶段。无明确结论时可能为 `null`。 |
| `max_face_area_ratio` | `number \| null` | 当前请求中可获得的人脸框面积占整张图片面积的最大比例，取值范围理论上为 `0 ~ 1`。值越大，表示人脸在图片中占比越大。若未检测到有效人脸框，则该字段可能为 `null`。 |
| `agface_fake_photo` | `object` | 第一阶段子能力的完整结果对象。该对象始终存在，因为本接口一定会先调用该子能力。 |
| `desktop_recapture_detect` | `object \| null` | 第二阶段子能力的完整结果对象。仅当第一阶段判定为真人且第二阶段实际执行时才有值，否则为 `null`。 |
| `desktop_recapture_executed` | `boolean` | 第二阶段是否实际执行。`true` 表示已调用 `desktop_recapture_detect`；`false` 表示未调用。 |
| `desktop_recapture_skip_reason` | `string \| null` | 第二阶段未执行时的原因说明。当前代码中常见值包括：`agface_fake_photo detected no face`、`agface_fake_photo already classified as fake photo`。若第二阶段已执行，则通常为 `null`。 |

#### `agface_fake_photo` 子结果字段

| 字段 | 类型 | 含义 |
|---|---|---|
| `capability` | `string` | 固定为 `agface_fake_photo`。 |
| `model_version` | `string` | 当前 runtime 中该 capability 的模型版本。 |
| `error_code` | `number` | 子能力执行状态码。`0` 表示成功。 |
| `error_msg` | `string` | 子能力错误信息。成功时通常为空字符串。 |
| `result.status` | `number` | 子能力业务状态。当前代码中 `1` 表示检测成功，`0` 常表示未检测到人脸。 |
| `result.result` | `number` | 该子能力的真假判定结果。`1` 表示翻拍/假体，`0` 表示真人。 |
| `result.confidence` | `number` | `agface_fake_photo` 输出的翻拍倾向置信度，数值越大表示越偏向翻拍。 |
| `result.face_bbox` | `number[4]` | 最大人脸框，格式为 `[x, y, w, h]`，单位为像素。 |

#### `desktop_recapture_detect` 子结果字段

| 字段 | 类型 | 含义 |
|---|---|---|
| `capability` | `string` | 固定为 `desktop_recapture_detect`。 |
| `model_version` | `string` | 当前 runtime 中该 capability 的模型版本。 |
| `error_code` | `number` | 子能力执行状态码。`0` 表示成功。 |
| `error_msg` | `string` | 子能力错误信息。成功时通常为空字符串。 |
| `result.is_fake` | `boolean` | 第二阶段是否判定为翻拍。 |
| `result.label` | `string` | 第二阶段的文本标签，通常为 `real` 或 `fake`。 |
| `result.score_real` | `number` | 第二阶段输出的真人分数。 |
| `result.score_fake` | `number` | 第二阶段输出的翻拍分数。 |
| `result.execution_provider` | `string` | 当前推理执行后端，例如 `cpu`、`cuda`、`tensorrt`。用于技术排查，不建议作为业务判断依据。 |

### 第三方接入建议

- 业务侧如需直接使用最终结论，优先读取：
  - `result.final_label`
  - `result.final_is_fake`

- 如需了解最终结论来自哪一阶段，读取：
  - `result.decision_source`

- 如需做图片质量或采集规范控制，可结合：
  - `result.max_face_area_ratio`

- 如需追踪底层判定详情，可查看：
  - `result.agface_fake_photo`
  - `result.desktop_recapture_detect`

## 5.3 应用层人脸属性检测

### 接口

```http
POST /api/v1/infer/detect_face_property
```

### 接口说明

该接口是面向第三方客户提供的**人脸属性综合分析接口**。

当前代码中，该接口会依次调用两个底层 capability：

1. `agface_face_property`
   - 输出姿态角、眼镜、口罩、闭眼、帽子、翻拍倾向等属性信息
2. `agface_barehead`
   - 输出是否裸头及其置信度

服务端会将两个子能力的结果做统一汇总，并返回：

- 汇总后的 `summary`
- 两个子能力的完整原始结果
- 最大人脸占图比例 `max_face_area_ratio`

### Form 参数

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `image` | file | 是 | 待分析图片，建议上传单人、正脸或近似正脸、清晰无遮挡的 JPG/PNG 图片 |

### 返回结果总体说明

该接口返回的是**属性综合结果**，并不只对应一个底层模型，因此：

- `capability = "detect_face_property"`
- `model_version = "composite"`
  - 表示这是应用层组合接口，不对应单一模型包版本

### 汇总逻辑说明

当前代码中的 `summary` 字段，是从两个子能力结果中直接抽取得到的轻量汇总：

- 来自 `agface_face_property`：
  - `angle`
  - `glasses`
  - `mask`
  - `eyeclosed`
  - `hat`
  - `fake`

- 来自 `agface_barehead`：
  - `is_barehead`
  - `barehead_confidence`

如果底层子能力未能返回对应字段，则这些汇总字段可能为 `null`。

### 请求示例

```bash
curl -X POST "http://localhost:8080/api/v1/infer/detect_face_property" \
  -F "image=@face.jpg"
```

| `agface_face_property` | `object` | `agface_face_property` 子能力的完整结果对象。 |
| `agface_barehead` | `object` | `agface_barehead` 子能力的完整结果对象。 |

#### `summary` 汇总字段

| 字段 | 类型 | 含义 |
|---|---|---|
| `summary.angle` | `string \| null` | 人脸姿态角信息。该字段来自 `agface_face_property`，通常表示 pitch / yaw / roll 等姿态角。 |
| `summary.glasses` | `number \| null` | 眼镜相关置信度。数值越大，表示越倾向于检测到眼镜。 |
| `summary.mask` | `number \| null` | 口罩相关置信度。数值越大，表示越倾向于检测到口罩。 |
| `summary.eyeclosed` | `number \| null` | 闭眼相关置信度。数值越大，表示越倾向于检测到闭眼状态。 |
| `summary.hat` | `number \| null` | 帽子相关置信度。数值越大，表示越倾向于检测到帽子。 |
| `summary.fake` | `number \| null` | 翻拍倾向相关置信度。该值来自 `agface_face_property` 内部的活体/翻拍分析结果。 |
| `summary.is_barehead` | `boolean \| null` | 是否裸头。`true` 表示判定为裸头，`false` 表示判定为非裸头，`null` 表示当前无法形成明确裸头结论。 |
| `summary.barehead_confidence` | `number \| null` | 裸头判定置信度。该值来自 `agface_barehead`。 |

#### `agface_face_property` 子结果字段

| 字段 | 类型 | 含义 |
|---|---|---|
| `capability` | `string` | 固定为 `agface_face_property`。 |
| `model_version` | `string` | 当前 runtime 中该 capability 的模型版本。 |
| `error_code` | `number` | 子能力执行状态码。`0` 表示成功。 |
| `error_msg` | `string` | 子能力错误信息。成功时通常为空字符串。 |
| `result.status` | `number` | 业务状态。当前代码中 `1` 表示分析成功，`0` 常表示未检测到人脸。 |
| `result.angle` | `string` | 人脸姿态角信息。 |
| `result.glasses` | `number` | 眼镜置信度。 |
| `result.mask` | `number` | 口罩置信度。 |
| `result.facew` | `number` | 人脸宽度占整张图片宽度的比例。该字段来自底层 capability，和 `max_face_area_ratio` 不是同一个概念。 |
| `result.eyeclosed` | `number` | 闭眼置信度。 |
| `result.hat` | `number` | 帽子置信度。 |
| `result.fake` | `number` | 翻拍倾向置信度。 |
| `result.face_bbox` | `number[4]` | 最大人脸框，格式为 `[x, y, w, h]`，单位为像素。 |

#### `agface_barehead` 子结果字段

| 字段 | 类型 | 含义 |
|---|---|---|
| `capability` | `string` | 固定为 `agface_barehead`。 |
| `model_version` | `string` | 当前 runtime 中该 capability 的模型版本。 |
| `error_code` | `number` | 子能力执行状态码。`0` 表示成功。 |
| `error_msg` | `string` | 子能力错误信息。成功时通常为空字符串。 |
| `result.status` | `number` | 业务状态。当前代码中 `1` 表示分析成功，`0` 常表示未检测到人脸。 |
| `result.result` | `number` | 裸头判定结果。`1` 表示裸头，`0` 表示非裸头。 |
| `result.confidence` | `number` | 裸头判定置信度。 |
| `result.face_bbox` | `number[4]` | 最大人脸框，格式为 `[x, y, w, h]`，单位为像素。 |

### 第三方接入建议

- 如果客户只关心“可直接展示的属性结果”，优先读取：
  - `result.summary`

- 如果客户需要更细粒度的底层判定值或排查能力差异，再读取：
  - `result.agface_face_property`
  - `result.agface_barehead`

- 如果客户需要控制采集规范，可结合：
  - `result.max_face_area_ratio`
  - `result.agface_face_property.result.facew`

- 其中：
  - `facew` 表示“人脸宽度占图片宽度的比例”
  - `max_face_area_ratio` 表示“人脸框面积占整张图片面积的比例”
  - 两者含义不同，请勿混用

## 6. agface 人脸比对接口

## 6.1 端到端人脸比对

### 接口

```http
POST /api/v1/agface/face_compare
```

### 作用

执行完整的人脸比对链路：

```text
detect -> crop -> feature_a -> feature_b -> cosine -> score
```

### Form 参数

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|---|---|---|---|---|
| `image_a` | file | 是 | - | 第一张人脸图片 |
| `image_b` | file | 是 | - | 第二张人脸图片 |
| `feature_model` | string | 否 | `agface_face_feature_glint512` | 特征提取 capability |
| `detector` | string | 否 | `agface_face_detect` | 检测 capability，传空或 `none` 表示跳过检测 |
| `margin_ratio` | float | 否 | `0.15` | 裁剪人脸时的边界扩展比例 |

### 允许的 `feature_model`

- `agface_face_feature_residual256`
- `agface_face_feature_glint512`
- `agface_face_feature_mobilenet256`

### 请求示例

```bash
curl -X POST "http://localhost:8080/api/v1/agface/face_compare" \
  -F "image_a=@face1.jpg" \
  -F "image_b=@face2.jpg" \
  -F "feature_model=agface_face_feature_glint512" \
  -F "detector=agface_face_detect" \
  -F "margin_ratio=0.15"
```

### 成功返回示例

```json
{
  "code": 0,
  "message": "success",
  "total_time_ms": 38.62,
  "feature_capability": "agface_face_feature_glint512",
  "detector_capability": "agface_face_detect",
  "faces": {
    "image_a": 1,
    "image_b": 1
  },
  "cosine": 0.823451,
  "score": 91.17,
  "dim": 512,
  "feature_a_sample": [0.012345, -0.032165, 0.004421],
  "feature_b_sample": [0.011928, -0.031991, 0.004770]
}
```

### 返回字段说明

| 字段 | 说明 |
|---|---|
| `cosine` | 原始余弦相似度 |
| `score` | 映射后的 0~100 分数 |
| `dim` | 特征维度 |
| `faces.image_a` | 第一张图检测到的人脸数 |
| `faces.image_b` | 第二张图检测到的人脸数 |

### 常见错误

#### feature capability 不在白名单

```json
{
  "detail": {
    "code": 1001,
    "message": "invalid feature_model; allowed: [...]"
  }
}
```

#### detector 或 feature capability 未加载

```json
{
  "detail": {
    "code": 2001,
    "message": "feature capability 'agface_face_feature_glint512' is not loaded",
    "loaded": ["agface_face_detect", "desktop_recapture_detect"]
  }
}
```

## 11. 常见问题

## 11.1 为什么 license 显示 `days_remaining = -1`，但接口仍能正常调用？

因为：

- `-1` 在当前系统中表示 **长期有效 / 无截止日期**
- 不表示过期

第三方如需展示，可按以下规则处理：

- `days_remaining == -1` -> 显示“长期有效”

## 11.2 如何知道某个 capability 当前是否真的能调用？

请调用：

```http
GET /api/v1/capabilities
```

只有该接口返回的 capability，才表示当前 runtime 已加载。

## 11.3 不同 capability 的 `result` 为什么结构不同？

因为：

- 平台统一使用 `POST /api/v1/infer/{capability}` 作为通用入口
- 每个 capability 的输出结构由自身业务决定
- 第三方应按具体 capability 文档解析 `result`
