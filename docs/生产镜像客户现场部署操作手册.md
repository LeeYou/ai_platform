## 1. 文档目的

本文档用于指导客户在现场服务器上完成 `ai-prod` 生产镜像的部署、启动、验收、升级与故障排查。

本手册以当前仓库中的真实实现为准，核心依据包括：

- `prod/Dockerfile`
- `deploy/docker-compose.prod.yml`
- `prod/docker-entrypoint.sh`
- `prod/web_service/resource_resolver.py`

当前生产镜像具备以下特点：

- 容器内默认服务端口为 `8080`
- 支持 **GPU 优先、CPU 回退**
- 支持 **宿主机挂载资源优先于镜像内置资源**
- 支持客户后续通过替换挂载目录中的模型、动态库、授权文件进行升级

---

## 2. 部署架构说明

客户现场生产环境通常只部署 **一个生产推理容器**：

- 容器名称：`ai-prod`
- 镜像名称：`agilestar/ai-prod:latest`
- 对外服务：HTTP REST API + Web 管理页面

服务内部结构如下：

```text
客户请求
  -> HTTP 服务 (FastAPI)
  -> Python ctypes 调用 libai_runtime.so
  -> capability 动态库 lib<capability>.so
  -> 模型目录 / manifest.json / 模型文件
```

### 资源优先级

当前生产容器采用如下资源优先级：

```text
宿主机挂载目录 > 镜像内置目录
```

即：

- 若宿主机挂载目录中存在对应资源，则优先使用挂载版本
- 若宿主机未提供对应资源，则回退使用镜像内打包的版本

这意味着：

- 首次交付可直接使用镜像内置资源启动
- 后续升级可只替换宿主机挂载目录中的资源，而无需重做整镜像

## 4. 宿主机环境准备

## 4.1 安装 Docker

若现场服务器尚未安装 Docker，请先安装：

```bash
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER
```

重新登录后验证：

```bash
docker --version
docker compose version
```

---

## 4.2 GPU 服务器额外准备

如果客户现场机器配有 NVIDIA GPU，并希望使用 GPU 推理，请完成以下检查：

```bash
nvidia-smi
```

若能正常输出 GPU 信息，再安装 NVIDIA Container Toolkit：

```bash
distribution=$(. /etc/os-release; echo $ID$VERSION_ID)
curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | \
  sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg
curl -s -L https://nvidia.github.io/libnvidia-container/$distribution/libnvidia-container.list | \
  sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' | \
  sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list
sudo apt-get update && sudo apt-get install -y nvidia-container-toolkit
sudo nvidia-ctk runtime configure --runtime=docker
sudo systemctl restart docker
```

验证容器内 GPU：

```bash
docker run --rm --gpus all nvidia/cuda:11.8.0-cudnn8-runtime-ubuntu22.04 nvidia-smi
```

如果客户现场为纯 CPU 服务器，可以跳过本节，容器会自动回退为 CPU 推理模式。

## 5. 宿主机目录初始化

生产容器默认使用宿主机根目录：

```text
/data/ai_platform
```

客户现场采用离线交付模式，不提供项目源码仓库，因此请直接在宿主机手动创建目录。

推荐执行：

```bash
sudo mkdir -p /data/ai_platform/{licenses,pipelines,models}
sudo mkdir -p /data/ai_platform/logs/prod
sudo mkdir -p /data/ai_platform/libs/linux_x86_64
sudo chmod 700 /data/ai_platform/licenses
sudo chmod -R 777 /data/ai_platform/logs
```

### 初始化后建议目录结构

```text
/data/ai_platform/
├── licenses/
├── pipelines/
├── logs/
│   └── prod/
├── models/
├── libs/
│   └── linux_x86_64/
└── data/
```

### 生产环境重点目录

| 目录 | 用途 |
|---|---|
| `/data/ai_platform/licenses/` | 存放授权文件 `license.bin` 与公钥 `pubkey.pem` |
| `/data/ai_platform/models/` | 存放各 capability 的模型包 |
| `/data/ai_platform/libs/linux_x86_64/` | 存放 Linux x86_64 动态库产物 |
| `/data/ai_platform/pipelines/` | 存放 pipeline JSON 配置 |
| `/data/ai_platform/logs/prod/` | 容器日志落地目录 |

### 权限建议

```bash
sudo chmod 700 /data/ai_platform/licenses
sudo chmod -R 777 /data/ai_platform/logs
```

### 自定义根目录说明

如果客户现场未使用默认根目录 `/data/ai_platform`，而是使用例如：

```text
/opt/ai_platform
```

则除了按相同结构手动创建 `/opt/ai_platform` 下的目录外，还必须同步修改 `docker-compose.prod.yml` 中的卷挂载路径。

例如，将：

```yaml
- /data/ai_platform/models:/mnt/ai_platform/models:ro
- /data/ai_platform/libs/${AI_ARCH:-linux_x86_64}:/mnt/ai_platform/libs:ro
- /data/ai_platform/licenses:/mnt/ai_platform/licenses:ro
- /data/ai_platform/pipelines:/mnt/ai_platform/pipelines:rw
- /data/ai_platform/logs/prod:/mnt/ai_platform/logs:rw
```

改为：

```yaml
- /opt/ai_platform/models:/mnt/ai_platform/models:ro
- /opt/ai_platform/libs/${AI_ARCH:-linux_x86_64}:/mnt/ai_platform/libs:ro
- /opt/ai_platform/licenses:/mnt/ai_platform/licenses:ro
- /opt/ai_platform/pipelines:/mnt/ai_platform/pipelines:rw
- /opt/ai_platform/logs/prod:/mnt/ai_platform/logs:rw
```

否则容器仍会去读取默认的 `/data/ai_platform/...` 路径，导致资源无法生效。

## 6. 客户资源摆放规范

## 6.1 授权文件

将以下文件放入：

```text
/data/ai_platform/licenses/
```

目录示例：

```text
/data/ai_platform/licenses/
├── license.bin
└── pubkey.pem
```

说明：

- `license.bin` 为客户授权文件
- `pubkey.pem` 用于校验授权签名
- 当前容器启动时优先读取挂载目录中的授权文件

如果挂载目录下没有 `pubkey.pem`，容器日志会提示：

```text
No pubkey.pem found — signature verification will be skipped
```

因此正式交付时建议始终提供 `pubkey.pem`。

---

## 6.2 模型目录

模型目录按 capability 分开管理。当前 runtime 读取模型时，要求路径满足：

```text
/data/ai_platform/models/<capability>/current/
```

并且 `current` 目录内必须存在：

- `manifest.json`
- 该 capability 所需模型文件

示例：

```text
/data/ai_platform/models/agface_fake_photo/current/
├── manifest.json
└── detection/
    ├── detection.param
    ├── detection.bin
    └── ...
```

说明：

- `current` 可以是真实目录，也可以是软链接
- 生产服务只认 `current` 这一层作为当前生效版本

---

## 6.3 动态库目录

动态库按 capability 管理。当前 `docker-compose.prod.yml` 中实际挂载的是：

```text
/data/ai_platform/libs/${AI_ARCH:-linux_x86_64}
    -> /mnt/ai_platform/libs
```

因此对 Linux x86_64 客户现场，推荐实际目录结构为：

```text
/data/ai_platform/libs/linux_x86_64/<capability>/current/lib/lib<capability>.so
```

例如：

```text
/data/ai_platform/libs/linux_x86_64/agface_fake_photo/current/lib/libagface_fake_photo.so
/data/ai_platform/libs/linux_x86_64/desktop_recapture_detect/current/lib/libdesktop_recapture_detect.so
```

### runtime 库

除 capability 动态库外，还需要提供 runtime 主库：

- `libai_runtime.so`

若采用 builder 产物的标准目录结构，也应放在 libs 目录中与其他能力库一起交付。

---

## 6.4 Pipeline 目录

若客户需要启用 pipeline 功能，请将 JSON 配置文件放入：

```text
/data/ai_platform/pipelines/
```

容器启动后将优先使用该挂载目录。

## 8. 生产启动配置

## 8.1 推荐使用 docker compose

建议将交付包中的 `docker-compose.prod.yml` 放到现场部署目录，例如：

```text
/opt/ai_platform_deploy/
```

### 推荐目录

```text
/opt/ai_platform_deploy/
├── docker-compose.prod.yml
└── .env
```

---

## 8.2 `.env` 配置示例

建议在 `docker-compose.prod.yml` 同目录创建 `.env` 文件：

```bash
cat > .env <<'EOF'
AI_ARCH=linux_x86_64
AI_ADMIN_TOKEN=please-change-to-a-strong-random-token
AI_MAX_INSTANCES=4
AI_ACQUIRE_TIMEOUT_S=30
AI_INFER_MAX_CONCURRENCY=16
AI_INFER_CONCURRENCY_TIMEOUT_SECONDS=30
AI_MAX_UPLOAD_BYTES=52428800
UVICORN_WORKERS=2
LOG_LEVEL=info
EOF
```

### 关键变量说明

| 变量 | 说明 |
|---|---|
| `AI_ARCH` | 挂载的动态库架构目录，默认 `linux_x86_64` |
| `AI_ADMIN_TOKEN` | 管理接口令牌，**生产必须修改** |
| `AI_MAX_INSTANCES` | Runtime/能力实例池大小，客户可按并发和显存调整 |
| `AI_ACQUIRE_TIMEOUT_S` | Runtime 获取能力实例的超时秒数 |
| `AI_INFER_MAX_CONCURRENCY` | HTTP 层并发请求上限，默认 `16` |
| `AI_INFER_CONCURRENCY_TIMEOUT_SECONDS` | HTTP 层等待并发槽位的超时秒数，默认 `30` |
| `AI_MAX_UPLOAD_BYTES` | 单次请求允许的最大上传大小，默认 `52428800`（50 MB） |
| `UVICORN_WORKERS` | Uvicorn worker 进程数，默认 `2` |
| `LOG_LEVEL` | 服务日志级别，例如 `debug`、`info`、`warning`、`error` |

### 参数调优建议

- **GPU 服务器**
  - 优先调 `AI_MAX_INSTANCES`
  - 再根据外部并发量评估是否提高 `AI_INFER_MAX_CONCURRENCY`

- **CPU 服务器**
  - 建议保守配置 `AI_MAX_INSTANCES`
  - 避免设置过高导致 CPU 抢占和响应抖动

- **大图/高分辨率场景**
  - 如客户现场确实需要传更大图片，可酌情提高 `AI_MAX_UPLOAD_BYTES`
  - 调整后建议重新做接口压测与稳定性验证

### 安全要求

`AI_ADMIN_TOKEN` 默认值在代码中是：

```text
changeme
```

因此客户现场部署时**必须**修改为高强度随机字符串。

## 9. 容器启动后的真实资源解析逻辑

当前容器入口脚本的实际逻辑如下。

## 9.1 GPU 检测

容器启动时会先执行：

- 若 `nvidia-smi` 可用，则按 GPU 优先模式运行
- 若 `nvidia-smi` 不可用，则自动切换为 CPU 模式

因此：

- GPU 服务器无需额外人工切换
- CPU 服务器也可直接启动

## 9.2 模型目录

当前逻辑：

- 若 `/mnt/ai_platform/models` 存在，则优先使用挂载模型目录
- 否则回退到镜像内 `/app/models`

## 9.3 授权目录

当前逻辑：

- 若 `/mnt/ai_platform/licenses` 存在，则优先使用挂载授权目录
- 默认读取：
  - `/mnt/ai_platform/licenses/license.bin`
  - `/mnt/ai_platform/licenses/pubkey.pem`
- 否则回退到镜像内：
  - `/app/licenses/license.bin`
  - `/app/licenses/pubkey.pem`

## 9.4 Pipeline 目录

当前逻辑：

- 若 `/mnt/ai_platform/pipelines` 存在，则优先使用挂载目录
- 否则回退到 `/app/pipelines`

## 10. 部署后验收

## 10.1 查看容器是否正常启动

```bash
docker ps | grep ai-prod
```

应看到容器状态为 `Up`。

---

## 10.2 健康检查接口

验证接口：

```bash
curl http://localhost:8080/api/v1/health
```

正常情况下应返回：

- `status`
- `capabilities`
- `license`
- `gpu_available`
- `runtime_initialized`

重点检查：

- `status` 是否为 `healthy`
- `runtime_initialized` 是否为 `true`
- `loaded_capability_count` 是否大于 `0`

---

## 10.3 能力列表检查

```bash
curl http://localhost:8080/api/v1/capabilities
```

用于确认：

- 当前已加载哪些 capability
- capability 对应版本是否正确

说明：

- 客户最终可调用能力，以该接口实际返回结果为准
- 仓库里存在 capability，不代表客户现场一定已经加载

---

## 10.4 授权状态检查

```bash
curl http://localhost:8080/api/v1/license/status
```

重点检查：

- `valid` 是否为 `true`
- `status` 是否为 `valid`
- `days_remaining`
  - 若为 `-1`，表示长期有效，并非过期

---

## 10.5 Swagger 页面

可通过浏览器访问：

```text
http://<服务器IP>:8080/docs
```

用于：

- 查看接口列表
- 在线调试接口
- 验证参数格式

---

## 10.6 业务接口联调建议

建议至少验证以下接口：

- `GET /api/v1/health`
- `GET /api/v1/capabilities`
- `GET /api/v1/license/status`
- `POST /api/v1/infer/{capability}`

如果现场已交付应用层接口，也可验证：

- `POST /api/v1/infer/detect_fake_photo`
- `POST /api/v1/infer/detect_face_property`

## 11. 升级与回滚

## 11.1 升级镜像

如需升级生产镜像：

1. 停止当前容器
2. 导入新镜像
3. 重新启动 compose

示例：

```bash
docker compose -f docker-compose.prod.yml down
docker load -i agilestar_ai_prod_vnext.tar
docker compose -f docker-compose.prod.yml up -d
```

如果模型、动态库、授权都仍挂载在宿主机目录中，则通常无需重新准备这些资源。

## 11.2 仅升级模型或动态库

若镜像不变，只升级资源：

- 替换 `/data/ai_platform/models/...`
- 或替换 `/data/ai_platform/libs/...`

然后重启容器或调用热重载接口。

### 重启容器方式

```bash
docker compose -f docker-compose.prod.yml restart prod
```

### 热重载接口方式

```bash
curl -X POST http://localhost:8080/api/v1/admin/reload \
  -H "Authorization: Bearer <AI_ADMIN_TOKEN>"
```

如需只重载单个 capability：

```bash
curl -X POST http://localhost:8080/api/v1/admin/reload/<capability> \
  -H "Authorization: Bearer <AI_ADMIN_TOKEN>"
```

## 11.3 回滚建议

建议客户现场保留：

- 上一版镜像 tar 包
- 上一版模型目录
- 上一版 libs 目录

这样在升级异常时可以快速恢复。

## 12. 常见问题排查

## 12.1 容器启动了，但 `health` 显示 `runtime_initialized=false`

排查顺序：

- 检查 `libai_runtime.so` 是否已正确交付
- 检查 `/data/ai_platform/libs/linux_x86_64/` 目录是否完整
- 检查 capability so 与 runtime 是否版本匹配
- 查看容器日志：

```bash
docker compose -f docker-compose.prod.yml logs -f prod
```

---

## 12.2 `capabilities` 返回为空

说明当前 runtime 没有成功加载 capability。

请检查：

- 模型目录是否存在 `manifest.json`
- `current` 是否指向正确模型目录
- `lib<capability>.so` 是否位于正确目录
- license 是否覆盖当前 capability

---

## 12.3 接口返回 `Capability version not licensed`

说明：

- capability 已被发现
- 但 capability 当前版本不满足 license 约束

排查建议：

- 检查模型目录中的 `manifest.json` 版本号
- 检查 capability so 对应版本
- 检查 `license.bin` 中的版本约束是否覆盖该版本

---

## 12.4 页面或接口显示 `days_remaining = -1`

这不是错误。

在当前系统中：

- `days_remaining = -1`
  - 表示**长期有效 / 无截止日期**

---

## 12.5 GPU 服务器未使用 GPU

检查：

- 宿主机 `nvidia-smi` 是否正常
- 是否安装了 NVIDIA Container Toolkit
- compose 中 GPU 保留配置是否保留
- 容器日志中是否显示检测到 GPU

---

## 12.6 修改了挂载资源但未生效

因为当前服务优先读取挂载目录，但运行中的 runtime 可能尚未重载资源。

建议执行以下任一方式：

- 重启容器
- 调用热重载接口

## 15. 结论

客户现场部署的关键点只有三项：

- **镜像已正确导入**
- **宿主机挂载目录中的授权、模型、动态库已正确摆放**
- **通过 `health`、`capabilities`、`license/status` 完成验收**

只要以上三项满足，生产镜像即可稳定运行，并支持后续通过宿主机挂载资源进行持续升级。
